# 订单与物流 Playbook

## 适用场景

用户询问：

- 订单是否成功。
- 订单到哪里了。
- 物流什么时候到。
- 支付是否完成。
- 订单是否能取消。
- 配送异常或延迟。

## 决策流程

1. 如果已有订单号或上下文中有订单 ID，先判断用户具体在问什么。
2. 查询订单总览和商品明细时调用 `order.lookup`。
3. 查询物流、快递、派送、预计送达时调用 `shipment.lookup`。
4. 查询支付、付款、扣款、交易流水时调用 `payment.lookup`。
5. 查询售后单、退款、退货、换货、维修状态时调用 `after_sales.lookup`。
6. 如果没有订单号，但用户已登录或有 `user_id`，可以先调用 `user.lookup` 或询问是否查询最近一笔订单。
7. 如果既没有订单号，也没有可用身份上下文，向用户请求订单号或安全核验信息。
8. MCP 返回成功后，用用户能理解的话解释当前状态和下一步。
9. MCP 返回权限不足时，不暴露内部权限规则，提示需要核验身份或转人工。
10. 物流延迟、丢件、签收异常等情况，必要时创建工单或转人工。

## 回复要点

- 明确当前状态：已支付、待发货、运输中、派送中、已签收、异常。
- 给出时间信息：预计到达时间、最近更新时间。
- 给出下一步：继续等待、联系配送、创建工单、转人工。
- 不承诺未经 MCP 确认的赔付、退款、改地址或取消结果。

## 工具

- `order.lookup`：查询订单总览和商品明细。
- `shipment.lookup`：查询物流、派送和预计送达时间。
- `payment.lookup`：查询支付状态和脱敏交易信息。
- `after_sales.lookup`：查询售后记录和处理状态。
- `user.lookup`：在缺少订单号但有身份上下文时查询可用订单提示。
- `ticket.create`：物流异常、用户投诉、需要后续跟进。
- `handoff.request`：高风险、权限冲突、强烈不满或用户要求人工。

## 示例

用户："帮我查订单 64575145823542368 的物流。"

动作：

1. 调用 `shipment.lookup`，参数包含 `order_id`、`user_id`、`tenant_id`、`brand_id`。
2. 根据工具结果回复，不要说“我将调用工具”，而是直接给出查询结果。
