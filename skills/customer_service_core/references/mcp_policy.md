# MCP 使用策略

## 原则

MCP 工具提供真实业务能力。skill 决定什么时候需要工具，但 MCP 决定某个操作是否被允许。

## 工具映射

- `knowledge.search`：静态产品、政策、FAQ、价保、售后规则和服务知识。
- `vision.analyze`：理解用户上传的图片，例如商品实物、故障照片、支付截图、物流截图、售后凭证或包装信息。
- `user.lookup`：用户身份上下文、会员信息、最近订单提示。
- `order.lookup`：订单总览，例如商品、订单状态、下单时间、金额摘要。
- `shipment.lookup`：物流与配送状态，例如承运商、运单号、派送进度、预计送达。
- `payment.lookup`：支付状态，例如付款状态、支付方式、交易号脱敏、退款支付侧状态。
- `after_sales.lookup`：具体订单的退货、换货、退款、维修、补发进度。
- `ticket.create`：创建投诉、反馈或跟进记录。
- `handoff.request`：转接人工客服。

## 选择顺序

1. 用户明确要人工，优先 `handoff.request`。
2. 用户强烈投诉但没有直接要求人工，可先 `ticket.create`，必要时转人工。
3. 用户给出订单号时，根据关键词选择：
   - 订单、状态、买了什么：`order.lookup`
   - 物流、快递、配送、派送、到哪：`shipment.lookup`
   - 支付、付款、扣款、流水：`payment.lookup`
   - 售后、退款、退货、换货、维修、补发：`after_sales.lookup`
4. 用户没有订单号但问“我的订单”，可先 `user.lookup` 获取最近订单提示，或直接追问订单号。
5. 用户问商品、参数、价格、价保、售后规则、FAQ，使用 `knowledge.search`。
6. 用户上传图片时，优先使用 `vision.analyze` 获取图片中可见内容。
7. 如果图片问题涉及商品是否在售、是否正品、价格、参数、型号真伪、外部品牌官方是否发布等事实，必须在 `vision.analyze` 后继续使用 `knowledge.search` 或商品目录核验。不能使用视觉模型或 LLM 自身知识判断。
8. 缺少必要信息时先澄清，不要调用无意义工具。

## 工具结果处理

MCP 成功时：

- 使用结构化数据保证准确性。
- 如果有 `display_summary`，可以使用，但必要时应改写成自然客服语言。
- 不要暴露原始工具 trace、内部 ID、检索分数、权限内部细节、数据库字段或堆栈错误。

MCP 失败时：

- `MISSING_ARGUMENT`：追问缺少的信息。
- `USER_NOT_FOUND`：说明当前账号信息不足，请用户重新登录或转人工。
- `ORDER_NOT_FOUND`：请用户核对订单号、账号或购买渠道。
- `PERMISSION_DENIED`：说明当前身份无法核验这笔订单归属，建议补充核验信息或转人工。
- `TOOL_UNAVAILABLE` 或系统异常：说明查询暂时失败，建议重试；连续失败时转人工。
- `vision_model_not_configured` 或 `vision_model_error`：说明当前暂时无法查看图片，建议重新上传、稍后重试或转人工。

## 图片与时效事实

视觉模型只能作为 OCR/图片内容提取器使用。

允许视觉模型输出：

- 图片中出现的文字。
- 图片中可见的商品外观、颜色、界面、故障现象。
- 包装、凭证、截图中的可见字段。

不允许视觉模型直接判断：

- 外部品牌是否已经发布某型号。
- 某型号是否真实存在。
- 某商品是否本店在售。
- 商品价格、库存、参数是否准确。
- 是否正品或是否支持保修。

这些问题必须通过 `knowledge.search`、商品目录、订单工具或人工核验回答。若知识库没有记录，应说“当前本店知识库/商品目录未查到该型号在售记录”，不要说“官方没有发布”。

## 副作用操作

对于创建工单、转人工等副作用：

- 确保用户意图明确。
- 使用可用身份上下文。
- 让 MCP 执行最终权限校验。
- 只反馈已经确认的结果。
